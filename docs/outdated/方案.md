# 自动解题系统方案

## 1. 目标定义

该系统的目标不是“做一个固定的四 Agent 框架”，而是：

1. 输入课程题目材料后，自动生成一份尽量像学生手写风格的 C++ 解答。
2. 用 `demo.exe` 作为标准行为参考，对生成程序进行自动测试与迭代修正。
3. 尽量隔离不同任务上下文，避免“解题 Agent”被测试、混淆、规则检查等信息污染。
4. 每次运行都生成独立工作区，保留过程痕迹，便于复盘。

推荐把项目定位为：

- 一个 Windows 上运行的本地自动化编排器
- 外挂多个 CLI Agent
- 先用单主控 + 多子任务的模式实现
- 未来再按需要扩展成真正的 multi-agent

## 2. 推荐实现路线

## 推荐结论

第一阶段不要直接做“4 个平级 Agent 自由协作”。

更稳妥的方案是：

1. 用一个 `orchestrator` 负责全流程编排。
2. 把“解题 / 造数据 / 校验 / 降专业度”做成独立 stage。
3. 每个 stage 可以调用不同 CLI Agent，但输入输出必须文件化。
4. stage 之间只通过工作区文件交换信息，不直接共享上下文。

这个结构比完全自由的 multi-agent 更容易调试，也更适合课程作业场景。

## 3. 建议的系统架构

### 3.1 核心模块

建议拆成以下模块：

1. `runner`
   - 接收输入参数
   - 创建本次运行目录
   - 调度各 stage
   - 记录日志和状态

2. `ingest`
   - 接收 PDF、`demo.exe`、可选测试数据
   - 抽取题目文字、图片引用、限制条件
   - 生成统一的题目描述文件

3. `solver`
   - 根据题目描述生成 `main.cpp`
   - 输出思路摘要、关键假设、复杂度分析

4. `testgen`
   - 基于题意生成补充测试数据
   - 如果存在给定数据文件，则调用 `get_input_data.exe` 读取各组样例

5. `evaluator`
   - 编译生成代码
   - 运行生成程序和 `demo.exe`
   - 使用 `txt_compare.exe` 比较输出
   - 汇总错误并反馈给 `solver`

6. `constraint_checker`
   - 检查是否违反题目限制
   - 检查是否用了禁用语法、禁用库、过于“超纲”的写法

7. `humanizer`
   - 在不改变行为的前提下做风格降级
   - 让代码更接近普通学生提交

8. `artifact_writer`
   - 汇总最终产物
   - 输出结果、日志、失败原因、候选代码版本

### 3.2 Agent 角色映射

文档中的 4 Agent 可以映射为：

1. 主解题 Agent
2. 测试数据 Agent
3. 校验 Agent
4. 风格调整 Agent

但推荐实现时不要把它们设计成长期对话实体，而是设计成：

- 独立 prompt 模板
- 独立输入文件
- 独立输出文件
- 由编排器按需调用

这样后续更换 Kimi CLI、Codex CLI、Claude Code 会更容易。

## 4. 目录设计

建议目录结构如下：

```text
auto_fuck_sj/
  docs/
    prompts/
      solver.md
      testgen.md
      checker.md
      humanizer.md
  tools/
    run_demo.bat
    run_compare.bat
    run_compile.bat
  src/
    orchestrator.py
    stages/
      ingest.py
      solver.py
      testgen.py
      evaluator.py
      checker.py
      humanizer.py
    adapters/
      cli_agent.py
      kimi_cli.py
      codex_cli.py
      claude_code.py
  workspaces/
    000001/
      input/
      extracted/
      testcases/
      candidates/
      outputs/
      logs/
      final/
```

## 5. 单次运行流程

推荐流程如下：

1. 创建 `workspaces/000001/`
2. 复制输入文件到 `input/`
3. 解析 PDF，生成：
   - `problem_statement.md`
   - `constraints.md`
   - `assets_manifest.json`
4. 调用 `solver` 生成首版 `main.cpp`
5. 编译并运行基础测试
6. 调用 `testgen` 生成补充测试
7. 对每组测试：
   - 调用 `demo.exe` 生成标准输出
   - 调用候选程序生成待测输出
   - 使用 `txt_compare.exe` 比较
8. 若失败，提取错误摘要反馈给 `solver` 生成下一版
9. 通过后运行 `constraint_checker`
10. 通过后运行 `humanizer`
11. 再次回归测试，确保混淆未破坏正确性
12. 输出最终结果到 `final/`

## 6. 建议的数据流

每个 stage 只读固定输入文件，只写固定输出文件。

例如：

- `solver`
  - 输入：`problem_statement.md`、`constraints.md`、历史失败摘要
  - 输出：`candidates/v1/main.cpp`、`candidates/v1/notes.md`

- `testgen`
  - 输入：题目描述、已有样例、限制条件
  - 输出：`testcases/generated_cases.json`

- `evaluator`
  - 输入：候选代码、测试集、`demo.exe`
  - 输出：`outputs/eval_report.json`

这样做的好处是：

1. 失败可复现
2. 不同 Agent 可替换
3. 后续可并行化
4. 可人工介入某一阶段

## 7. PDF 处理建议

需求里提到 PDF 可能包含文本和图片。这里建议分层处理：

### 第一层：能提文本就先提文本

先提取：

- 可复制文本
- 页码
- 代码块
- 样例输入输出

### 第二层：图片单独记录

把图片裁剪或导出后，生成：

- 图片文件
- 图片所在页码
- 图片上下文文字

### 第三层：必要时整份 PDF 直接交给 Agent

如果题目排版复杂、公式多、图片强相关，允许把 PDF 文件路径直接作为 Agent 输入的一部分。

结论是：

不要把“完美 PDF 解析”作为前置条件。只要能形成一个足够好的题目摘要，就可以先进入解题流程。

## 8. 编译与评测策略

Windows 环境下建议把编译、运行、比较都包装成脚本，避免主程序里塞太多平台细节。

### 8.1 编译

建议支持至少一种稳定编译器：

- `g++`
- 或 Visual Studio `cl`

MVP 阶段建议先固定一种，避免过多兼容逻辑。

### 8.2 评测

对于每个测试点：

1. 准备输入文件
2. 运行 `demo.exe < input.txt > expected.txt`
3. 运行 candidate 程序得到 `actual.txt`
4. 执行：

```bat
txt_compare.exe --file1 actual.txt --file2 expected.txt --trim right --ignore_blank --display normal
```

`--trim right` 与 `--ignore_blank` 可以作为默认宽松策略，但建议做成可配置。

### 8.3 反馈闭环

失败后不要把完整日志一股脑喂给解题 Agent，而是生成结构化摘要，例如：

- 第几组测试失败
- 输入规模特征
- 首个不一致行
- 可能原因分类
  - 边界条件遗漏
  - 输入解析错误
  - 输出格式问题
  - 算法逻辑错误

这样更容易让 Agent 迭代修正。

## 9. “限制条件检查”建议

这个模块不要只做字符串搜索，建议分两层：

### 9.1 规则层

简单规则可直接检查：

- 是否包含禁用头文件
- 是否使用 `vector` / `string` / STL
- 是否出现递归
- 是否出现类、模板、lambda 等超纲语法

### 9.2 风格层

由 Agent 结合题目限制做主观判断：

- 写法是否过于高级
- 命名是否过于工程化
- 是否明显不像课程作业

这个阶段不应直接改代码，只负责出报告。

## 10. “降低专业度”建议

这个模块风险较高，因为容易把正确代码改坏。建议明确边界：

允许做的事：

- 改变量名为更普通的风格
- 减少过度模块化
- 去掉过于正式的注释
- 把过强的抽象改成较直接写法

不建议做的事：

- 主动替换核心算法
- 大规模重排控制流
- 为了“像学生”故意引入坏味道或潜在 bug

推荐方式是：

1. 先让 `humanizer` 输出修改计划
2. 再生成新代码
3. 强制重新全量测试

## 11. 推荐的 MVP 边界

第一版只做以下能力就够了：

1. 支持单题运行
2. 支持 PDF + `demo.exe` + 可选数据文件输入
3. 支持生成一版代码并自动编译
4. 支持调用 `demo.exe` 和 `txt_compare.exe` 做自动比对
5. 支持失败后反馈再生成 1 到 3 轮
6. 支持最终做一次风格调整并回归测试

先不要做：

1. 多题批处理
2. 真正并行的多 Agent 会话
3. 复杂 UI
4. 远程服务化
5. 过度通用的插件系统

## 12. 技术选型建议

推荐主控语言使用 Python，原因：

1. 编排 CLI 简单
2. 处理文件和日志方便
3. 后续接入 PDF 解析、JSON、模板容易
4. Windows 上落地成本低

推荐主控形式：

- 一个命令行入口，例如：

```bash
python -m src.orchestrator run ^
  --pdf problem.pdf ^
  --demo demo.exe ^
  --data hw_data.txt
```

## 13. 为什么不建议一开始做“完全自由协作”的多 Agent

原因很直接：

1. 调试困难
2. 成本不可控
3. 上下文污染严重
4. 输出稳定性差
5. 很难复现某次失败

更好的做法是：

“编排器主导，Agent 执行子任务，文件作为唯一中介。”

这已经能满足“多个 Agent 真正协作”的要求，而且更工程化。

## 14. 后续可演进方向

MVP 跑通后，可以继续加：

1. 多候选代码并行生成，择优进入评测
2. 针对 WA / TLE / RE 的不同修复 prompt
3. 基于题型分类选择不同 solver 模板
4. 自动检测图片题、表格题、数学公式题
5. 记录成功案例，做题型经验复用

## 15. 我建议你现在采用的落地方案

如果现在就开工，我建议按下面顺序推进：

1. 先实现 Python 编排器
2. 固定使用 Codex CLI 跑通全流程
3. 固定使用 Windows 上的 `g++`
4. 打通 `demo.exe` + `txt_compare.exe` 自动评测
5. 让评测产物全部落盘，便于人工复查
6. 再加测试数据生成
7. 最后再加风格调整和多 Agent 适配

这个顺序能最快验证核心价值：自动产出正确代码。

## 16. 已确认事项

根据当前补充信息，第一版方案进一步收敛为：

1. 目标优先级是“最可能先跑起来”，不是先追求架构完整性。
2. 首个 Agent CLI 固定为 Codex。
3. Windows 下默认编译器固定为 `g++`。
4. 最终会由人工再用 VS2022 做一次复检，因此系统内不必第一阶段兼容 `cl`。
5. `demo.exe` 使用标准输入和标准输出工作。
6. 评测时应把标准答案输出和候选程序输出都写入文件，再交给 `txt_compare.exe` 比较。
7. 题目整体难度不会太高，因此第一版重点应放在流程稳定性，而不是复杂题型策略。
8. 代码风格目标不是“故意写差”，而是“不过分工程化，但保留基本科班素养”。

推荐的风格边界：

- 变量名不能退化成 `a`、`b`、`c`、`d`
- 不使用拼音变量名
- 不堆砌工程化抽象
- 保持正常的函数拆分、命名和注释水平
- 看起来像认真完成课程作业的学生代码，而不是竞赛模板或工业代码

## 17. 第一版最小实现建议

既然目标是“最可能先跑起来”，那第一版应进一步压缩：

1. 只支持单题
2. 只支持 Codex
3. 只支持 `g++`
4. 只做文本优先的 PDF/题面提取
5. 只保留一个主解题 Agent，加一个测试/校验子任务
6. 最多迭代 3 轮修复
7. 最终输出：
   - `main.cpp`
   - 编译日志
   - 测试日志
   - `demo.exe` 输出文件
   - candidate 输出文件
   - compare 结果

也就是说，第一版其实可以不是“完整多 Agent 系统”，而是：

一个带子任务调用能力的单主控编排器。

只要工作区、评测闭环、反馈修复闭环、风格收敛这四件事跑通，就已经满足核心目标。

## 18. 待确认问题

以下问题会直接影响方案落地，建议优先确认：

1. 题目限制里是否经常出现“不能用 STL / 不能用某些语法”这类硬性规则？
2. `get_input_data.exe` 和 `txt_compare.exe` 是否已经保证在目标 Windows 机器上可直接调用？
3. 你希望系统最终输出只有一个 `main.cpp`，还是还要附带过程报告、测试报告、失败版本归档？

## 19. 结论

这件事最合适的形态，不是“纯概念 multi-agent”，而是：

一个以工作区为中心的 Windows 本地编排器。

它调用多个 CLI Agent 执行解题、造数据、校验和风格调整，并通过 `demo.exe` + `txt_compare.exe` 形成自动反馈闭环。

这是最容易先做成、也最容易后续扩展的方案。
