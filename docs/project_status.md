# 项目状态与目标

## 项目定位

`auto_fuck_sj` 是一个 Windows 本地运行的最小可用产品 (MVP)，可以用 Codex 求解课程作业型 C++ 子题，使用 `demo.exe` 验证行为，并把完整求解过程保存在编号工作区中。

当前主链路已具备：

- 读取题目文件（PDF/文本）
- 创建编号工作区
- 加载提供的分组测试用例
- 调用 Codex 生成候选 C++ 文件
- 用 `g++` 编译
- 运行候选代码和 `demo.exe`
- 用 `txt_compare.exe` 比较输出
- 保存所有中间产物

## 已解决的关键问题

- 当 PDF 文本无法提取时，求解器以前会猜题目提示语和输出格式
- 带中文反馈的重试以前会因为 `codex exec` 的 stdin 没强制 UTF-8 而失败
- 候选代码输出以前会因为 Windows 本地中文输出编码未对齐而与 `demo.exe` 不一致
- 评测时的文本重编码以前会破坏比对输入
- 不相关的分组测试用例以前会泄漏到当前子题运行中
- 求解器重试以前会收到过大的原始比对日志，而不是紧凑失败摘要
- `5-b16` 子题验证以前会忽略 `demo.exe` 所需的参数标志

## 开发目标

### 主要目标

构建一个可靠的 Windows 本地编排器，用 Codex 求解课程作业型 C++ 子题，使用 `demo.exe` 验证行为，并把完整求解过程保存在独立的工作区中。

### 近期目标

- 保持多个已验证子题上的求解链路稳定
- 让 `demo.exe` 的子题选择变得显式且可重复
- 降低求解器对猜测格式的依赖
- 提升重试时失败反馈的质量
- 改进对课程特定禁用规则的硬约束检测
- 让回归验证可以稳定地从已检入规范重复执行

### 中期目标

- 在 Windows 上补强 PDF 提取能力，优先接入 `pdftotext`
- 减少必须手工知道 demo 参数这类交接知识的依赖
- 让编码行为在更多作业上都显式且可重复
- 在不破坏正确性的前提下改进人性化处理
- 用更多真实课程样例扩充回归覆盖

### 质量目标

- 在 `5-b15` 和 `5-b16 --sub1..--sub4` 上稳定重复通过端到端验证
- 常见作业的平均重试次数较低
- 工作区产物保持可读、可排障
- 在最终输出前就暴露约束违规
- 最终代码保持正确，同时仍像正常学生作业

### 当前非目标

- 大规模多题批处理
- 完整的自由式多 Agent 架构
- GUI 或 Web 界面
- 超出当前 Windows `g++` 目标之外的广泛工具链兼容性
